# himitu-tool コーディング規約 (AIコーディング・Vanilla想定)

本プロジェクトは、**生成AI（CopilotやCursorなど）を活用して機能実装を高速化すること**と、**ビルドなしでレンタルサーバーで動くこと（Vanilla環境）**を前提としています。
AIがコンテキストを読み取りやすく、かつ推論中にエラーやブレを起こしにくい（＝ハルシネーションを減らす）ためのルールを定めています。

## 1. AIが理解しやすい構造づくり（最重要）

* **影響範囲の局所化（スコープの分離）**
  ツールを跨いでファイルを行き来するとAIが文脈を見失いやすくなります。各ツールの開発は、`tool/[ツール名]/` のフォルダ内で**完全に完結**するようにします。
* **命名規則の一貫性**
  AIはパターン認識が得意です。ツール別のファイル・ID・プレフィックスには必ず「ツール名」を含めます。
  * フォルダ名： `tool/timer/`
  * ファイル群： `index.html`, `timer.css`, `timer.js`
  * ルートID： `<div id="timer-app">`（CSSやJSでのスコープをこれに絞る）

```text
tool/
  ├── css/style.css       # サイト全体共通（ヘッダー/フッターなど。AIに読ませる文脈として必要時以外は触らない）
  ├── js/script.js        # サイト全体共通スクリプト
  └── [ツール名]/           # 個別ツール（AIを適用するメイン領域）
      ├── index.html      # ツールのDOM構造（必ず #ツール名-app でラップする）
      ├── [ツール名].css    # 固有スタイル（ #ツール名-app プレフィックスを必ず付ける）
      └── [ツール名].js     # 固有ロジック（ type="module" で読み込む ）
```

## 2. JavaScriptの規約 (Vanilla JS)

* **モジュール化 (`<script type="module">`)**
  グローバル変数の衝突を避けるため、HTMLからの読み込みは必ず `type="module"` とします。これにより、AIがどのスコープで変数を処理すべきかのブレを防ぎます。
* **状態（State）の単一管理**
  UIの切り替えやデータの保持について、DOMから値を直接読み取るのではなく、JS内にシンプルな状態管理オブジェクトを持つ方がAIは処理（コードの修正・リファクタリング）を正確に行えます。
  ```javascript
  // AIにとって予測・修正しやすいステート管理
  const state = {
    isRunning: false,
    timeLeft: 0,
  };
  ```
* **コメントをプロンプトにする**
  関数の前や複雑なロジックには、AI（Copilotなど）への指示（プロンプト）を兼ねた明確な日本語コメントを残します。
  ```javascript
  // タイマーの残り時間を hh:mm:ss フォーマットに変換してDOMに描画する
  function renderTime(seconds) { ... }
  ```

## 3. CSSとHTMLの規約 (Vanilla)

* **CSSスコープの絶対要件**
  AI生成コードによって他のページ（ツール）が崩れる事故を防止するため、スタイルはIDまたは固有のプレフィックス配下に閉じ込めます。
  ```css
  /* GOOD: AIが生成しても他機能に影響が出ない */
  #timer-app .display-panel { display: flex; }
  ```
* **セマンティックなHTMLとID付け**
  AIはHTMLの構造（IDとクラス）をベースにJSのDOM操作コードを生成します。Jsから操作される要素には `id="timer-start-btn"` のように、後からAIが特定しやすい一意なIDやクラスを付与してください。
* **状態の管理はデータ属性（`data-*`属性）で**
  UIのオンオフなどを持たせる際は、`data-active="true"` などを用います。AIに対し「アクティブ状態のCSSを書いて」と頼んだ際に一貫したコードが出力されやすくなります。

## 4. 拡張機能・静的検証（AIと一緒に使うと便利なもの）

AIが書き出したコードの一貫性を担保するために、手元のVS Code等で以下の導入を推奨しますが、本番環境には影響しません。
* **Prettier**: AIが生成したコードのインデントの揺れなどを一瞬で統一します。
* **Live Server**: ローカル開発用の簡易サーバー。`type="module"`でのJS検証に必須となります。
# コーディング規約 (Coding Guidelines)

## 1. リンク（ルーティング）に関するルール
当プロジェクトはサブディレクトリ（例: /tool/）へデプロイされる前提で動作しています。そのため、Next.jsの asePath: '/tool' 設定を利用しています。
内部リンクを記述する際は、必ず **Next.jsの <Link> コンポーネントを使用してください**。

### 良い例（OK）
Next.jsの Link を使うと、自動的に href に /tool/ が付与されるため、正しく遷移できます。
`	sx
import Link from 'next/link';

// トップページへのリンク
<Link href="/">トップへ戻る</Link>

// 別ツールへのリンク
<Link href="/timer">タイマーへ</Link>
`

### 悪い例（NG）
通常の <a> タグを使用すると、ドメインのルート（/）に飛んでしまい、別のサイトにアクセスしてしまいます。
`	sx
//  サイトのルート（hmts.jp/）に飛んでしまう
<a href="/">トップへ戻る</a>

//  hmts.jp/timer に飛んでしまう
<a href="/timer">タイマー</a>
`

> **注意:** 外部サイトへのリンク（例: https://hmts.jp などドメインまで含んだ絶対URL）については、引き続き <a> タグを使用して問題ありません。
